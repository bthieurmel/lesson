Documenter (1/2)

roxygen2 : documentation au-mme niveau que la fonction dans le script R, gnration automatique de la documentation LaTeX prsente dans le dossier man et dition du fichier NAMESPACE !

Le plus simple : placer le curseur au-niveau de la fonction et faire Code -> Insert roxygen Skeleton ou bien utiliser le raccourci clavier associ Ctrl+Alt+Shift+R.

Les balises obligatoires

  • @param : pour les arguments
  • @return : pour le rsultat
  • @examples : pour les exemples
  • @export : fonction principale du package, et donc exporte / visible pour lutilisateur

Et aussi

  • \code{character} pour une syntaxe de code, \link[pkg]{function} pour des liens
  • pour les exemples : \dontrun{}, \dontshow{}, \donttest{}
  • @rdname pour une aide partage entre plusieurs fonctions

Gestion des dpendances

Appeler une fonction externe proprement avec la syntaxe package::fonction ne suffit pas pour bien dfinir et grer nos dpendances, comme le check vous le rappelle gentillement, avec dans notre exemple :

-- R CMD check results -------------------- demopck 0.0.0.9000 ----
Duration: 25.4s

> checking dependencies in R code ... WARNING
  '::' or ':::' import not declared from: 'numbers'

En effet, les dpendances doivent tre dfinies 3 endroits :

  • dans le documentation roxygen2 de la fonction, avec les balises @import ou @importFrom
  • dans le fichier NAMESPACE. Ce dernier scrit automatiquement quand on re-gnre la documentation
  • et dans le fichier DESCRIPTION

Dpendances : dans la documentation

La premire tape est donc dajouter dans la documentation des fonctions les informations sur les packages dpendants

  • @import : importation de tout le package dpendant au chargement de notre package. A utiliser si on utilise beaucoup de fonctionnalits dun autre package
#'@import package1 package2 package3
  • @importFrom : importation dun sous-ensemble de fonctions dun package dpendant au chargement de notre package. A privilgier.
#'@importFrom package fonction1 fonction2 fonction3

Le fichier NAMESPACE sera alors automatiquement complt quand on mettra jour la documentation :

import(shiny)
import(yaml)
importFrom(DT,DTOutput)
importFrom(DT,datatable)

Dpendances : dans DESCRIPTION

Mettre des import et des importFrom ne suffit pas pour satisfaire le check Il faut galement rajouter ces dpendances dans le fichier DESCRIPTION. Cest possible de lditer manullement, cependant il est assez psychorigide sur le format attendu. Il est conseill dutiliser usethis pour cela avec use_package() qui sen chargera donc pour nous :

Il est trs important de mettre en place des tests unitaires. Ces tests seront minima lancs chaque check du package, et on pourra galement configurer des lancements automatiques avec un gestionnnaire de code ( chaque commit par exemple).

Le package testthat est l pour a !

Initialisation :

  • cration dun dossier tests avec
    • un fichier testthat.R ( ne pas modifier en gnral)
    • et un dossier testthat dans lequel on va insrer nos tests sous la forme de scripts R

Bonnes pratiques : Couvrir lensemble de la fonction avec des tests ET rajouter un nouveau test chaque bug rencontr et corrig

Excution : Build -> More -> Test package, Ctrl+Shift+T ou devtools::test()

Testthat : syntaxe

  • criture de scripts R, sauvegarder dans tests/testthat. Convention de nommage : test-*.R

Principales fonctions :

  • context("infos") : Information sur les tests qui suivent
  • test_that("info", {tests}) : Dfinition dun bloc de test
  • expect_equal() : galit avec une tolrance de prcision, expect_identical() : galit stricte
  • expect_false() | expect_true() : retourne effectivement TRUE ou FALSE
  • expect_message() | expect_warning() | expect_error() : affichage de message, warning ou erreur
  • et pleins dautres!
context("Nombres premiers")

test_that("Bons rsultats", {
  expect_false(is_premier(1))
  expect_true(is_premier(3))
})

Aller plus loin : le dossier inst/

Le dossier inst permet de rajouter dautres ressources notre package, comme par exemple :

  • des fichiers plats utilises dans des exemples ou dans les fonctions, de configuration
  • des scripts de tests / dmo
  • une application shiny

Lors de linstallation du package, le dossier inst/ sefface et lensemble de son contenu est alors prsent la racine du package. On utilise alors la fonction system.file pour y accder.

exemple :

  • nous avons une application shiny dans inst/app/
#' Launch app
#' @import shiny # plus les autres dpendances de l'appli
run_demo_app <- function(...) {
  shiny::runApp(appDir = system.file("app", package = "demopck"), ...)
}
