Les packages existants

Il existe au minimum un package qui permet de se connecter chaque type de bases de donnes, par exemple :

  • RMySQL : base de donnes MySQL
  • RPostgres : base de donnes PostgreSQL ( prioriser vs le package plus ancien RPostgreSQL, qui nest plus maintenu et moins performant)
  • ROracle : base de donnes Oracle
  • RSQLServer : base de donnes MS SQL Server
  • RSQLite : base de donnes SQLite
  • RJDBC : connexion diffrents types de bases de donnes via du Java
  • RODBC : connexion diffrents types de bases de donnes via du ODBC
  • mongolite : base de donnes NoSQL mongodb
  • .

DBI : syntaxe usuelle de connexion (1/2)

Le package DBI propose une interface commune dutilisation des diffrentes bases de donnes.

Les autres packages comme RMySQL ou RPostgres servent alors essentiellement fournir le driver permettant la connexion la base de donnes correspondante.

La syntaxe usuelle dutilisation est la suivante :

  1. ouverture de la connexion avec dbConnect()
  2. requtage avec dbGetQuery(), avec le passage directement dune requte SQL
  3. fermeture de la connexion avec dbDisconnect()

DBI : syntaxe usuelle de connexion (2/2)

require(DBI)
require(RPostgres)

# ouverture d'une connexion
con <- dbConnect(RPostgres::Postgres(), # RMySQL::MySQL()
                 host = "myserver", 
                 port = 123, 
                 dbname = "database", 
                 user = "benoit", 
                 password = "security")

# excution d'une requte et rcupration du rsultat
res <- dbGetQuery(con, "SELECT * FROM mytable")

# Fermeture de la connexion
dbDisconnect(con)

DBI : principales fonctions

  • dbConnect() & dbDisconnect() : connexion / dconnexion la base
  • dbListTables() & dbListFields() : rcupration des tables disponibles et des champs prsents
  • dbGetQuery() : envoi dune requte SQL retournant des donnes (SELECT ...) et rcupration du rsultat
  • dbReadTable() & dbWriteTable() : lecture / criture dune table dans sa globalit
  • dbSendQuery() : excution dune requte
    • suivie dun dbFetch() si on souhaite rcuprer le rsultat
    • et dans tous les cas de dbClearResult()
    • va nous permettre notamment dinsrer des lignes / supprimer des tables,

DBI : dbConnect

Le premier argument obligatoire de dbConnect est donc le driver permettant de se connecter la base de donnes. Ce driver tant en gnral fourni pour le biais dun autre package.

Les principaux paramtres de connexion sont ensuite les suivants :

  • user : nom dutilisateur
  • password : mot de passe
  • host : adresse ip ou nom de lhte hbergeant la base
  • port : numro du port ouvert de la base
  • dbname : nom de la base de donnes

Quelques exceptions cependant dans le nom des arguments, comme par exemple lors de la connexion une base MS SQL

DBI : dbGetQuery

Lutilisation de la fonction dbGetQuery avec le passage directe dune requte SQL est ensuite la faon la plus simple dinterroger la base de donnes :

# connexion  une base temporaire SQLite in-memory
con <- dbConnect(RSQLite::SQLite(), ":memory:")

# ecriture d'une table
dbWriteTable(con, "mtcars", mtcars)

# excution d'une requte et rcupration du rsultat
res <- dbGetQuery(con, "SELECT * FROM mtcars WHERE cyl = 4");head(res, n = 2)
##    mpg cyl  disp hp drat   wt  qsec vs am gear carb
## 1 22.8   4 108.0 93 3.85 2.32 18.61  1  1    4    1
## 2 24.4   4 146.7 62 3.69 3.19 20.00  1  0    4    2
# dconnexion
dbDisconnect(con)

ATTENTION bien respecter les rgles associes la base de donnes, comme lutilisation de simple quote ou la casse des caractres

DBI : dbSendQuery

dbSendQuery excute une requte sur la connexion. Elle ne retourne pas par dfaut le rsultat, ce qui possible avec un dbFetch(), et il faut dans tous les cas finir par un dbclearResult() pour librer la mmoire.

# connexion  une base temporaire SQLite in-memory
con <- dbConnect(RSQLite::SQLite(), ":memory:"); dbWriteTable(con, "mtcars", mtcars)
# excution de la requte
rs <- dbSendQuery(con, "SELECT * FROM mtcars WHERE cyl = 4"); rs
## <SQLiteResult>
##   SQL  SELECT * FROM mtcars WHERE cyl = 4
##   ROWS Fetched: 0 [incomplete]
##        Changed: 0
# rcupration du rsultat
data <- dbFetch(rs); head(data, n = 2)
##    mpg cyl  disp hp drat   wt  qsec vs am gear carb
## 1 22.8   4 108.0 93 3.85 2.32 18.61  1  1    4    1
## 2 24.4   4 146.7 62 3.69 3.19 20.00  1  0    4    2
# nettoyage & dconnexion
dbClearResult(rs) ; dbDisconnect(con)








SECURISER LES INFORMATIONS DE CONNEXION

Via une fentre utilisateur ?

Dans RStudio avec rstudioapi::askForPassword(), ou avec getPass::getPass()

# ouverture d'une connexion
con <- dbConnect(RPostgres::Postgres(),
                 host = "myserver", 
                 port = 123, 
                 dbname = "database", 
                 user = "benoit", 
                 password = rstudioapi::askForPassword("Database password"))

avec les options dans R ?

Dans un script priv et non partag, dfinir des options de connexion :

options(database_userid = "myuserid")
options(database_password = "mypassword")

Puis dans le script principal et partag :

# ouverture d'une connexion
con <- dbConnect(RPostgres::Postgres(),
                 host = "myserver", 
                 port = 123, 
                 dbname = "database", 
                 user = getOption("database_userid"), 
                 password = getOption("database_password"))

avec des variables denvironnement ?

Mme principe quavec les options, sauf que les variables denvironnement doivent tre dfinies directement sur la machine, ou bien dans le fichier .Renviron prsent la racine.

# .Renviron
database_userid = "username"
database_password = "password"

Puis dans le script principal et partag :

# ouverture d'une connexion
con <- dbConnect(RPostgres::Postgres(),
                 host = "myserver", 
                 port = 123, 
                 dbname = "database", 
                 user = Sys.getenv("database_userid"), 
                 password = Sys.getenv("database_password"))

dans un fichier YAML ?

exemple.yml :

# Liste de configuration pour une base de donnes
db:
  host : 10.244.36.68
  port : 5432
  uid : user
  pwd : pwd
  dbname : database

Dans R, on charge ensuite le fichier avec la fonction yaml.load_file :

require(yaml)
conf <- yaml.load_file("exemple.yml")
con <- dbConnect(RPostgres::Postgres(),
                 host = conf$db$host, port = conf$db$port, 
                 dbname = conf$db$dbname, user = conf$db$uid, 
                 password = conf$db$pwd)

avec le package config ?

Le package config facilite la gestion denvironnements spcifiques, en se basant simplement sur une fichier config.yml.

  • par dfaut, config cherchera ce fichier dans le rpertoire de travail courant. Mais nous pouvons galement passer le chemin du fichier.

Plus dinformations : https://cran.r-project.org/web/packages/config/vignettes/introduction.html

Concatnation avec paste ?

Ds lors que lon souhaite dvelopper des requtes paramtrables, il est facile et tentant dutiliser simplement la fonction paste pour crer la requte finale.

cyl_subset <- 4
# excution de la requte
res <- dbGetQuery(con, 
  paste(
    "SELECT * FROM mtcars WHERE cyl = ", 
    cyl_subset
  )
)

Cela amne potentiellement une faille de scurit car, sans contrles pralables des arguments passs, lutilisateur peut trs bien insrer une reqete dangereuse et commettre une attaque

Il est donc conseill dutiliser des requtes dtes paramtrables, ce qui possible de plusieurs faons :

  • en utilisant dbSendQuery() et dbBind()
  • ou le package glue et la fonction glue_sql()

dbSendQuery & dbBind (1/2)

  1. cration dune requte avec dbSendQuery, en utilisant des ? pour le/les paramtre(s)
  2. utilisation de dbBind pour passer la/les valeur(s)
  3. rcupration avec dbFetch
  4. Appels multiples possibles
# un argument simple
con <- dbConnect(RSQLite::SQLite(), ":memory:")
dbWriteTable(con, "mtcars", mtcars)

simple_bind <-  dbSendQuery(con, "SELECT * FROM mtcars WHERE cyl = ?")
dbBind(simple_bind, list(4))
head(dbFetch(simple_bind), n = 2)
##    mpg cyl  disp hp drat   wt  qsec vs am gear carb
## 1 22.8   4 108.0 93 3.85 2.32 18.61  1  1    4    1
## 2 24.4   4 146.7 62 3.69 3.19 20.00  1  0    4    2

dbSendQuery & dbBind (2/2)

# plusieurs arguments simples
con <- dbConnect(RSQLite::SQLite(), ":memory:")
dbWriteTable(con, "mtcars", mtcars)

multiple_bind <-  dbSendQuery(con, "SELECT * FROM mtcars WHERE cyl = ? AND am = ?")
dbBind(multiple_bind, list(4, 0))
head(dbFetch(multiple_bind), n = 2)

Inconvnient majeur : : impossible de passer des arguments plus complexes, comme des ensembles de valeurs pour loprateur SQL IN par exemple

glue_sql (1/2)

  1. cration de la requte avec glue_sql, en utilisant des {nom_var} pour le/les paramtre(s)
  2. rcupration directement avec dbGetQuery
# un argument simple
con <- dbConnect(RSQLite::SQLite(), ":memory:")
dbWriteTable(con, "mtcars", mtcars)

simple_glue <- glue::glue_sql(
  "SELECT * FROM mtcars WHERE cyl = {val_cyl}", 
  val_cyl = 4, # passage des variables
  .con = con # et de la connexion
)

simple_glue

  "SELECT * FROM mtcars WHERE cyl IN ({val_cyl*}) AND am = {val_am}",

  val_am = 0,  
  .con = con
)

DBPLYR

dbplyr, cest quoi ?

Le package dbplyr permet dutiliser la syntaxe du package dplyr, mais directement sur des bases de donnes. Le code R sera alors convertit automatiquement en une requte SQL, et cette requte sera excuter le moment venu quand on souhaitera rcuprer le rsultat dans R. Il fonctionne date avec les connexions / bases de donnes suivantes :

  • RMySQL
  • RPostgreSQL et RPostgres
  • RSQLite
  • odbc
  • bigrquery (Googles )

Manipuler les donnees avec dplyr

Les bases de dplyr : slection des colonnes

select permet de slectionner des colonnes dun tableau de donnes

  • les noms sans quote, avec un - devant si on veut plutt enlever certaines colonnes, ou en utilisant loprateur : pour une plage
mtcars %>% select(cyl, disp) # uniquement cyl et disp
mtcars %>% select(-cyl, -disp) # tout sauf cyl et disp
mtcars %>% select(cyl:wt) # de cyl  wt
  • en utilisant des chanes de caractres et any_of ou all_of
mtcars %>% select(any_of(c("cyl", "disp"))) # dyl et disp
  • avec des expressions rgulires et les fonctions starts_with, ends_width, contains ou matches
mtcars %>% select(starts_with("c")) # les colonnes commenant par c

Les bases de dplyr : filtre sur les lignes

filter permet de filtrer les lignes conserver

  • en passant une ou plusieurs expressions R qui retournent des boolans (>=, ==, !=, %in%, )
  • dans le cas de plusieurs expressions, elles seronts combines avec le & logique
mtcars %>% filter(am == 0) # renvoie les lignes avec am = 0
mtcars %>% filter(am == 0 | am == 1) # am = 0 ou am = 1
mtcars %>% filter(am == 0 & vs == 1) # am = 0 et vs = 1
mtcars %>% filter(am == 0, vs == 1) # am = 0 et vs = 1
  • et on peut utiliser toutes les fonctions R disponibles
mtcars %>% filter(mpg == max(mpg))
  • utilisation des noms de colonnes en caractres ? avec .data, qui fait rfrence lensemble des donnes (list)
var = "am" ; value = 0
mtcars %>% filter(.data[[var]] == value)

Les bases de dplyr : ordonner les donnes

arrange permet dordonner les donnes

  • Par dfaut par ordre croissant, en utilisant le nom de la colonne sans quote
mtcars %>% arrange(mpg) # tri par mpg croissant
mtcars %>% arrange(am, mpg) # am croissant, puis mpg croissant
  • utilisation des desc pour un tri dcroissant
mtcars %>% arrange(desc(mpg)) # tri par mpg desc
mtcars %>% arrange(desc(am), mpg) # am desc, puis mpg croissant
  • .data avec des chanes de caractres
var <- "mpg"
mtcars %>% arrange(desc(.data[[var]]))

Les bases de dplyr : crer des nouvelles colonnes

mutate permet de crer une ou plusieurs nouvelles colonnes. Syntaxe : nom (avec ou sans quote) = expression

mtcars %>% mutate(wt_sq = wt ^ 2) # ajout de wt au carre
mtcars %>% mutate("wt_sq" = wt ^ 2) # idem
  • plusieurs colonnes la suite, les colonnes tant crer dans lordre, elles sont donc disponibles si besoin pour la cration des colonnes suivantes
mtcars %>% mutate(wt_sq = wt ^ 2, wt_sq_sup10 = wt_sq > 10)
  • et toujours .data avec des chanes de caractres!
var <- "wt" ; mtcars %>% mutate(wt_sq = .data[[var]] ^ 2)
  • conditions multiples ? case_when
mtcars %>% mutate(am_vs = case_when(
  am == 0 & vs == 0 ~ "Nothing", am == 0 & vs == 1 ~ "Vs only",
  am == 1 & vs == 0 ~ "Am only", am == 1 & vs == 1 ~ "Am & Vs",
  TRUE ~ "Other" # valeur par dfaut
))

Les bases de dplyr : agrgation par sous-population (1/2)

La fonction group_by va nous permettre de dfinir les sous-population sur lesquelles nous allons pouvoir effectuer des oprations

  • En enchanant sur un mutate, nous calculerons par exemple une nouvelle variable pour chaque sous-population et le rsultat sera affect lensemble du jeu de donnes initial
# on enchane les oprations avec %>%
# le group_by doit plac avant le mutate
mtcars_mod <- mtcars %>% group_by(cyl) %>%
  mutate(mean_mpg = mean(mpg)) %>%
  select(mpg, cyl, mean_mpg)

Les bases de dplyr : agrgation par sous-population (2/2)

  • En enchanant sur un summarize, on pourra calculer des indicateurs sur nos sous-populations et rcuprer directement le tableau agrg (summarize peut tre utiliser sur lensemble des donnes, sans group_by)

Les bases de dplyr : autres fonctions utiles

  • slice() : slectionner des lignes du tableau selon leur position
  • relocate() : changer lordre des colonnes, rename() : changer le noms des colonnes
  • sample_n() et sample_frac() : slectionner un chantillon alatoire
  • lead() et lag() : crer des variables dcales
  • distinct() : conserver uniquement les observations uniques

fusion de tables

  • inner_join() : les lignes prsentes dans le premier jeu de donnes x ET le seconde jeu de donnes y sont conserves dans la table finale, par rapport aux clefs dfinies

  • full_join() : les lignes prsentes dans le premier jeu de donnes x OU le seconde jeu de donnes y

  • left_join() : les lignes prsentes uniquement dans x

  • right_join() : les lignes prsentes uniquement dans y
